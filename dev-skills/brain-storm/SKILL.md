# 头脑风暴

基于项目上下文的**通用答疑**：主代理编排探索，子代理读代码与文档并返回探索报告，主代理据此给出**有依据**的回答。不默认产出 PRD/SPEC、不改代码；用户明确要求时再切换对应任务。

主代理与子代理的协作、说明、最终回复**一律使用中文**（路径、标识符、协议字段等除外）。

---

## 目的

回答用户关于本项目的**各类问题**，而非仅做「方案可行性」裁决。典型诉求包括：

- 弄清现状：某功能怎么工作、某模块负责什么、数据从哪来到哪去
- 定位问题：报错可能原因、行为为何与预期不符、该看哪些文件
- 对比与取舍：多种实现路径的差异、复用现有能力 vs 新建模块
- 影响评估：改 X 会波及哪些模块、有无既有约定或测试约束
- 方向探讨：在动手写 PRD/SPEC 之前，先澄清背景与可选路径

结论须基于探索报告中的证据；**禁止**仅凭常识、猜测或只读需求文本作答。

---

## 适用场景（范围宜宽）

| 类型 | 示例 |
|------|------|
| **理解现状** | 「登录流程怎么走？」「这个 hook 谁在调用？」 |
| **代码导航** | 「改国际化应该看哪些文件？」「配置项在哪定义？」 |
| **行为与逻辑** | 「为什么会触发两次？」「这个状态什么时候重置？」 |
| **架构与边界** | 「A 模块和 B 模块的边界在哪？」「事件是怎么广播的？」 |
| **方案探讨** | 「能不能用现有 X 做 Y？」「有几种做法，各有什么代价？」 |
| **影响与风险** | 「动这个 API 会影响谁？」「有没有现成测试覆盖？」 |
| **历史与文档** | 「以前迭代里讨论过类似方案吗？」「spec 里怎么写的？」 |
| **排错线索** | 「从日志/现象看，最该先查哪条链路？」 |
| **落地前摸底** | 写 PRD/SPEC 前，先弄清现状与可选路径（仍在本 skill 内答疑，不自动落盘） |

简单问题可拆 **1–2** 个子探索任务；跨模块、架构级或因果链复杂的问题宜拆 **2–4** 个并行子任务。

---

## 使用说明

1. **先理解问题**（可简短复述）：
   - 用户想知道什么、打算用答案做什么
   - 隐含约束（环境、版本、性能、兼容、工期等）
   - **AskQuestion 时序**：默认先完成探索再提问；**仅当**缺少决定性信息且**无法界定**探索范围时，可先 `AskQuestion` 只问 **1** 个聚焦问题
   - `AskQuestion` 不可用时：在回复中列出该单一聚焦问题，请用户补充
2. **在回复之前，必须先完成项目探索**（不可跳过）：
   - 主代理直读 `.apm/kb/docs/` 下相关文档、或用户指定路径（仅此步可由主代理直接做；**无 APM 时勿尝试 apm 命令 fallback**）
   - APM 可用时可额外 `apm read`、`apm kb search --q "<关键词>"`
   - **阅读代码与文档须派遣多个子代理**（见「探索阶段」），主代理 **不得** 自行深入读代码替代子代理
   - 主代理汇总各子代理的 **探索报告**，再组织对用户的回复
3. **探索后澄清**（如仍有关键缺口）：
   - 使用 `AskQuestion` 只问 **1** 个聚焦问题，须基于探索结论
   - `AskQuestion` 不可用时：在回复「仍不确定 / 待补充」中列出该单一聚焦问题
4. **组织回复**（见「回复结构」）：
   - 直接回应用户问题；可按问题类型选用章节，不必强行「可行/不可行」
   - 每条关键论断须对应探索报告中的证据（路径、符号、配置、文档）
   - 存在多种答案或不确定时，分点说明并标注置信度与待验证项
5. **边界**：
   - 本 skill **不**默认写 PRD、SPEC、任务拆分、不改代码
   - 探索报告 **仅供本次答疑**；**不构成**后续实现阶段跳过 impl 子代理的依据
   - 用户明确要求写文档或改代码时，按用户指定切换任务，**本 skill 内不自动展开**
6. **记忆**（**APM 可用时**，且结论值得跨会话复用时须更新）：
   - `dynamic`：本次问题、探索范围、结论摘要
   - `persist`：可长期复用的结论（如「模块 X 的入口在 …，不负责 Y」）

## 探索阶段（回复之前必做）

探索由 **主代理编排、子代理执行**；主代理依据探索报告作答，不替代子代理读代码。

### 主代理职责

1. 根据用户问题，将探索面拆为 **1–4 个互不重叠的子任务**（按复杂度；能并行则并行）
   - 例：用户问「改 agent 生命周期会影响哪些界面」→ 拆为「生命周期核心模块」「前端订阅与 UI」「相关测试与文档」
2. **并行** 派遣 **1–4** 个 readonly 探索子代理（`Task`，`subagent_type: explore`，`readonly: true`），**同步等待** 全部返回
3. 汇总各报告，识别一致结论、矛盾与缺口；有缺口时补派聚焦探索（仍由子代理执行）；**补派上限 2 轮**
4. 基于汇总结果撰写对用户的回复

### 子代理派遣规范

| 项 | 值 |
|----|-----|
| 工具 | `Task`，`subagent_type: explore` |
| readonly | **true** |
| 并行 | 1–4 个（简单题 1–2，复杂题 2–4），同步等待 |
| 失败 | 见「失败处理」 |

### 主代理禁止

- 自行深入阅读源码、遍历目录、追踪调用链（应写入子代理 prompt）
- 未等探索子代理返回即回答用户或下结论

### 探索子代理须返回（探索报告）

每条报告须为中文，结构如下：

```text
1）探索范围：本任务覆盖的模块/路径/文档
2）发现摘要：与用户问题直接相关的事实（按重要性排序）
3）证据列表：文件路径、符号、配置项、日志点、文档段落（须可核对）
4）关联与影响：上下游、依赖、同类实现（如有）
5）未弄清 / 矛盾点：子代理无法确认的部分
6）对主代理的建议：补探索方向、或需用户补充的信息（如有）
```

### 探索维度参考

| 探索项 | 何时纳入 |
|--------|----------|
| 入口/路由/CLI/配置 | 用户问功能从哪接入、开关、环境差异 |
| 核心模块与调用链 | 用户问职责、流程、谁调用谁 |
| 数据流与存储 | 用户问状态、持久化、序列化、边界 |
| API / 事件 / 协议 | 用户问契约、兼容性、消息格式 |
| 测试、CI、构建 | 用户问如何验证、回归范围、构建约束 |
| 知识库 / 历史迭代 | 用户问既往决策、PRD/spec、变更记录 |
| 错误处理与日志 | 用户问排错、异常路径、可观测性 |

探索深度与问题匹配：点名文件/符号的窄问题可单任务深挖；开放性问题宜多任务覆盖面。

### 探索子代理 prompt 模板

```text
【语言要求】
- 全程使用中文；路径、符号、命令可保留原文

请以 readonly 模式探索下列范围，回答主代理汇总所需的项目事实（不修改任何文件）。

仓库：<REPO_PATH>
用户问题：<USER_QUESTION>
探索范围：<SCOPE，如「agent 生命周期状态机与对外事件」>
探索目的：<WHAT_TO_FIND，如「理清状态迁移与 UI 订阅点」>

请先列出将阅读的文件/目录，再按「探索报告」结构返回。
禁止修改任何文件；禁止仅凭猜测下结论，找不到的写入「未弄清」。
```

## 回复结构

按用户问题选用章节，**不必**每次填满；**不要**把通用答疑强行套成「可行性判决」。

```markdown
## 回答摘要

<直接回应用户问题的 1–3 句话>

## 依据

- `<路径或符号>`：<从探索报告摘录的关键事实>
- …

## 说明

<展开解释：流程、因果、模块关系、多种理解等>

## 可选路径 / 建议（如适用）

1. <若用户在做决策：可选做法与利弊>
2. <若用户要动手：建议先读/先改哪里>
3. <若需验证：小实验、需用户拍板的点>

## 仍不确定 / 待补充（如适用）

- <探索未能闭合的点；需用户补充的上下文>
```

### 表述原则

- **问什么答什么**：解释型问题以说明为主；决策型问题再比较选项
- **证据优先**：先事实后推断；推断须标明依据
- **承认边界**：探索未覆盖处明确写出，不装作确定
- **避免越权**：不擅自写 PRD/SPEC、不改代码，除非用户明确要求并切换到对应 skill

---

## 失败处理

**子代理失败**（超时、资源等）：

1. 短暂停顿后重试一次
2. 仍失败：主代理可手工 readonly 探索，标注「手工探索」；**禁止**以违规自读代码替代子代理流程
3. 仍无法在证据下作答：在回复中说明缺口与已尝试范围，勿猜测性下结论
