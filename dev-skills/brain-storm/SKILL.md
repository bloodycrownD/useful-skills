# 头脑风暴

轻量探索与答疑：先摸清现状，再给出**有依据**的判断与建议。不默认产出 PRD/SPEC；用户明确要求时再落盘文档。

## 适用场景

- 「能不能做 X？」「Y 方案行不行？」
- 实现路径、技术选型、架构取舍
- 对现有模块/接口/流程的疑惑
- 在写 PRD 或 SPEC 之前，先快速验证方向

## 使用说明

1. **先理解问题**（可简短复述）：
   - 用户想达成什么
   - 隐含约束（性能、兼容、工期、团队习惯等）
   - 若关键信息缺失且会显著影响结论，用 `AskQuestion` 只问**一个**聚焦问题
2. **在给出结论之前，必须先探索项目**（不可跳过）：
   - 有 APM 时：`apm read`，`apm kb search --q "<关键词>"`
   - 列出可能涉及的文件、模块、接口、配置、测试
   - 阅读关键实现，确认现状、边界与已有类似能力
   - 优先 `Task`（`subagent_type: explore`）或等效方式并行检索
   - **禁止**仅凭常识或需求文本下结论
3. **形成判断**（见下方「回复结构」）：
   - 明确结论：可行 / 部分可行 / 不可行 / 需更多信息
   - 每条关键论断须对应代码或文档证据（路径 + 要点）
   - 不可行或风险高时，给出替代方案或降级路径
4. **边界**：
   - 本 skill **不**默认写 PRD、SPEC、任务拆分；用户要正式文档时，引导使用 `requirement-review` 或 `design-proposal`
   - 用户只要快速答疑时，直接在对话中回复即可
5. **可选记忆**（项目已用 APM 且结论值得跨会话复用时）：
   - `apm dynamic write --stdin`（多行推荐）：本次问题、探索范围、当前结论
   - `apm persist write --stdin` 或 `replace`：已确认的可复用结论（如「模块 X 不支持 Y，应走 Z」）

## 探索指引

| 探索项 | 目的 |
|--------|------|
| 入口/路由/CLI/配置 | 功能从哪接入、开关在哪 |
| 相关模块与服务 | 是否已有类似实现可复用 |
| 数据流与边界 | 类型、存储、上下游、第三方依赖 |
| 测试与文档 | 既有约定、回归风险 |
| 知识库/历史迭代 | 是否讨论过同类方案 |

探索深度与问题复杂度匹配：简单疑问读 1–3 个关键文件即可；架构级问题需更广的模块与依赖梳理。

## 回复结构

按此模板组织最终回复（可省略无内容的章节）：

```markdown
## 结论

**<可行 | 部分可行 | 不可行 | 需更多信息>** — 一句话摘要

## 依据

- `<路径或符号>`：<从代码/配置/文档观察到的关键事实>
- …

## 分析

<为何得出上述结论：与现状的契合点、冲突点、主要风险>

## 建议

1. <首选路径或下一步>
2. <备选或降级方案>
3. <若需验证：建议的小实验、需用户拍板的点>

## 开放问题

- <仍不确定、需用户补充或需 spike 验证的项>
```

### 结论用词

| 结论 | 何时使用 |
|------|----------|
| **可行** | 现有架构/代码可直接支撑或改动面清晰可控 |
| **部分可行** | 核心能做，但有明显缺口、代价或范围需收窄 |
| **不可行** | 与现状硬冲突、缺关键依赖、或成本明显不合理 |
| **需更多信息** | 探索后仍缺决定性输入（业务规则、环境、外部系统等） |

## 执行检查清单

- [ ] 已理解用户问题与约束
- [ ] 已探索代码/配置/文档（含 APM 检索，若可用）
- [ ] 结论有具体路径或符号作为依据，非空泛推断
- [ ] 已说明风险与替代建议（如适用）
- [ ] 未擅自展开为 PRD/SPEC，除非用户明确要求
