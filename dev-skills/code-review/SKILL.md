# 代码评审（Code Review）

## 用途

对**已实现代码**做结构化评审，输出分级问题、must-fix 与明确结论。适用于：

| 场景 | 典型触发 | 评审范围 |
|------|----------|----------|
| **独立 CR** | 「帮我 review 这次改动」「看看这个 PR」 | `diff` / `explore` |
| **阶段 CR** | `code-dev-loop` 某 phase 完成后 | `stage` |
| **合并前 CR** | 全部实现与收尾完成后 | `final` |

**执行方式**：

- **独立 CR**：主代理或子代理均可，读取 diff 后按本 skill 输出报告
- **`code-dev-loop` pipeline 内**：`cr-stage` / `cr-final` **必须**由 **readonly 评审子代理**执行（实现者不得自审，compact 模式下 impl 是主代理时亦然）
- 默认 **不改代码**；用户明确「review 并修」时方可修复

评审说明与报告**一律使用中文**（路径、符号、命令等除外）。

---

## 开始前

1. 明确**评审范围**：`BASE_SHA`…`HEAD_SHA`、文件列表、或 phase 名
2. 有 spec/PRD 时读取；无 spec 时以 PR 描述、commit、用户说明为需求来源，并在报告中标注「无 spec，依据为 …」
3. **列出将阅读的文件**后再下结论；禁止未读代码即评审
4. 有 APM 时：`apm read`、`apm kb search --q "<关键词>"`

---

## 评审模式

| 模式 | 代号 | 通过结论 | 何时使用 |
|------|------|----------|----------|
| 改动集 / PR | `diff` | **通过 / 需修改 / 阻塞** | 独立 CR、任意增量改动 |
| 探索式 | `explore` | **通过 / 需修改 / 阻塞** | 无明确 diff，按模块/路径 Review |
| 阶段 | `stage` | **phase-ready / not phase-ready** | `code-dev-loop` 单 phase 后 |
| 合并前 | `final` | **merge-ready / not merge-ready** | `code-dev-loop` 全部完成 + 收尾后 |

`diff` / `explore` 与 `stage` / `final` 使用**同一套检查维度**；差异在范围大小与结论词。`final` 额外强制 **G. 收尾与可合并性**。

---

## 检查维度

按改动类型选用；**A–F 为常规必查**，**H–K 按上下文选用**（在报告中注明「本次未适用」即可）。

### A. 需求与 Spec 符合性

- spec / PRD / 用户诉求中的变更点是否已实现
- **逐步对照**：spec「详细实现步骤」逐步列出 **已实现 / 部分 / 未实现 / 用 fallback 替代**（fallback 须标 P1，未经用户确认不得 phase-ready）
- **测试用例矩阵**：spec「测试用例」章节有编号（如 T-W1）时，须逐条对照 **已测 / 未测 / 不适用**
- 验收标准是否可由代码 + 测试支撑
- API、事件、配置、行为与契约一致
- **范围蔓延**：改动是否超出 stated scope；遗漏或过度实现均须标注

### B. 正确性与健壮性

- 逻辑是否正确；边界条件（空值、空集、零、极值、并发序）
- 错误处理是否完整；错误信息是否可诊断
- 资源释放：连接、文件句柄、订阅/监听器、定时器
- 回滚/幂等：失败重试、重复提交是否安全（若场景相关）
- 类型与契约：TypeScript 禁滥用 `any`；公开接口类型是否准确

### C. 代码质量与可维护性

| 项 | 检查要点 |
|----|----------|
| **风格与约定** | 与项目一致；lint/format 配置；目录与分层习惯 |
| **DRY** | 重复逻辑、复制粘贴、重复类型/常量 |
| **命名** | 表意清晰；与项目术语一致 |
| **复杂度** | 过长函数、过深嵌套、上帝类/模块 |
| **职责边界** | 模块是否单一职责；耦合是否过紧 |
| **死代码** | 未使用导入、不可达分支、注释掉的旧代码 |

### D. 安全与敏感数据（有用户输入/鉴权/IO 时必查）

- 注入：SQL、命令、路径遍历、XSS（视栈而定）
- 鉴权与授权：是否绕过权限；敏感操作是否校验身份
- 密钥/Token：是否硬编码、误提交、日志泄露
- 输入校验：外部数据是否校验与规范化
- 依赖：新增依赖是否必要、版本是否可信（大改动时）

### E. 性能与可扩展（热点路径或数据层改动时查）

- 明显 N+1、无界循环、大对象重复分配
- 缓存滥用或缺失（仅标注明显问题，不做 premature optimization）
- 数据库/文件 IO 是否在循环内（若适用）

### F. 中文注释与文档

与 `code-dev-loop` 一致（本项目硬性要求）：

- 关键分支、不变量、非显而易见逻辑须有**中文**注释
- 公开 API 须有文档注释（JSDoc/JavaDoc/docstring 等，正文中文）
- 英文说明性注释 → **P0**
- spec 要求更新的 README/CHANGELOG/模块头是否已更

### G. 测试与验证

- 改动是否有对应测试（或书面豁免理由）
- 测试是否断言**行为**而非实现细节
- 是否覆盖失败路径与边界（不只 happy path）
- 阶段/合并前：验证命令是否已跑且通过（主代理或报告须提供证据）

### H. 兼容性与迁移（涉及 API/数据/schema 时查）

-  breaking change 是否文档化、是否有迁移步骤
- 默认值与旧行为是否兼容
- Feature flag / 配置开关是否合理

### I. 可观测性（涉及服务端/异步/分布式时查）

- 关键失败是否有足够日志（无敏感信息泄露）
- 日志级别是否合理；是否留调试级噪音

### J. 前端 / UI（仅前端改动时查）

- 无障碍基本项（语义、焦点、aria 等，按项目标准）
- 国际化：文案是否走 i18n（项目若采用）
-  loading/error/empty 状态是否处理

### K. 收尾与可合并性（`final` 必查；独立 CR 合并前建议查）

以下任一项未完成 → **not merge-ready** / 独立 CR 判 **阻塞**：

- [ ] 调试代码、`console.log`、临时代码已清除
- [ ] 无关改动、误提交文件已 revert
- [ ] spec 要求的文档已更新
- [ ] lint / format 已通过（项目有则必须）
- [ ] 无未闭合 P0/P1
- [ ] 工作区干净（无未提交杂项）
- [ ] 提交信息可读（可建议 squash，不强制）

## 严重度与闭合标准

| 级别 | 含义 | 典型例子 |
|------|------|----------|
| **P0** | 必须修复才能通过 | 功能错误、spec 冲突、安全漏洞、调试残留、缺关键测试 |
| **P1** | 必须修复才能通过（本项目） | 严重风格偏离、明显 DRY 违反、缺中文注释/公开 API 文档 |
| **P2** | 建议，可遗留 | 命名微调、小重构建议、非阻塞优化 |

| 模式 | 通过条件 |
|------|----------|
| `diff` / `explore` | P0 = 0 且 P1 = 0 → **通过**；有 P0 → **阻塞**；仅 P1 → **需修改** |
| `stage` | P0 = 0 且 P1 = 0，且 spec 步骤无未实现/未确认 fallback，测试用例无遗漏 → **phase-ready** |
| `final` | 同上（全量）+ **K 节全部 ✅** → **merge-ready** |

独立 CR 时 P2 可单独列为「建议改进」，不阻塞 **通过**。

## 输出格式

```text
【评审模式】diff | explore | stage | final
【评审范围】<SHA 范围 / 文件列表 / phase 名>
【需求依据】<spec 路径 | PRD | 用户说明>

0）阅读范围
- <将审查的文件/目录列表>

1）需求符合性
- 实现步骤矩阵：<Step N → 已实现|部分|未实现|fallback + 证据>
- 测试用例矩阵（若 spec 有编号）：<T-xx → 已测|未测|N/A>
- 完成矩阵：<变更点 → 已实现/部分/未实现 + 证据>
- 范围蔓延 / fallback（若有，须标 P0/P1）

2）正确性与健壮性
- <P0|P1|P2>：<问题> — <文件:行或符号> — <建议>

3）代码质量
- ...

4）安全（若适用，否则写「本次未涉及」）
- ...

5）性能（若适用）
- ...

6）注释与文档
- ...

7）测试与验证
- ...

8）兼容性 / 可观测性 / UI（按适用填写，不适用则跳过）

9）收尾项（final 或合并前建议）
- <逐项 ✅/❌>

10）Must-fix 汇总（P0/P1）
- ...

11）建议改进（P2，可选）
- ...

12）结论：<通过 | 需修改 | 阻塞 | phase-ready | not phase-ready | merge-ready | not merge-ready>
    理由：<一句话>
```

## 执行检查清单

- [ ] 已明确模式与范围，并列出将读文件
- [ ] 已读 spec/PRD 或替代需求来源
- [ ] 已输出实现步骤矩阵；spec 有测试编号时已输出测试用例矩阵
- [ ] A–G 已覆盖；H–K 已按适用性检查或标注未适用
- [ ] 每条 P0/P1 含位置与可操作建议
- [ ] 结论与 must-fix 一致；未用「合并后再改」作为通过理由
- [ ] 默认未修改代码（除非用户授权修复）

## 与 code-dev-loop 的关系

- `cr-stage` / `cr-final`：**强制** readonly 评审子代理 + 本 skill（主代理不可替代）
- 评审须输出 **实现步骤矩阵** 与 **测试用例矩阵**（spec 有编号时），防止「测了一部分就宣称 ready」
- 独立 CR（`diff` / `explore`）：不依赖 DAG，主代理可直接评审
- 本 skill 定义**评什么、怎么判**；`code-dev-loop` 定义**何时评、mode、Context Bundle、循环修复**

---

## 备注

- 评审深度与改动大小匹配：单行 fix 不必强行套满 H–K
- 无 spec 时须在报告中声明依据，避免按猜测/requirement 幻觉评审
- 与安全/合规相关的 P0，即使 spec 未写也应列出
