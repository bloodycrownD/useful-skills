# Bug 修复

## 总流程

```text
探索定位 → 子代理修复（同步等待）→ 补充 prd.md / spec.md → 验证与收尾
```

主代理负责编排；**实现类工作须派子代理 inline 完成**，主代理不得跳过子代理自行改代码（资源故障见「失败处理」）。

## 路径约定

| 产物 | 路径 |
|------|------|
| Bug PRD | `.apm/kb/docs/Iterations/<需求名称>/bugs/<bug名称>/prd.md` |
| Bug SPEC | `.apm/kb/docs/Iterations/<需求名称>/bugs/<bug名称>/spec.md` |

- `<需求名称>`：所属迭代（与 `Iterations/` 下目录一致）；无法确定时询问用户
- `<bug名称>`：简短 kebab-case 标识（如 `login-timeout-on-refresh`）
- 两个文档写完后均执行 `apm kb index rebuild`

## 使用说明

### 1. 探索定位（不可跳过）

在修复或派子代理之前完成：

- 执行 `apm read`，`apm kb search --q "<bug 关键词/模块/需求名称>"`
- 读取父级 `.apm/kb/docs/Iterations/<需求名称>/prd.md`（及已有 `spec.md` 若存在）
- 复现或从用户描述还原：**现象、预期、实际、触发条件、影响范围**
- 探索相关代码：入口、调用链、配置、测试、近期变更
- 优先 `Task`（`subagent_type: explore`）并行检索
- 产出「探索结论」：疑似根因、影响文件、修复方向、回归风险
- 关键信息缺失时，用 `AskQuestion` 只问**一个**聚焦问题（复现步骤、环境、是否阻塞等）

**阶段完成**：更新 `dynamic`（bug 名称、现象、探索结论、待修复项）与 `persist`（根因要点、关键模块映射）。

### 2. 子代理修复（同步等待）

探索结论足以动手时：

1. **分支安全闸**：禁止在 `main` / `master` 直接改；切 `fix/<bug名称>` 或独立 worktree
2. 工作区须干净；位于保护分支时先切换
3. 向子代理派发 **单个 inline 修复任务**（`Task`，按任务选 `generalPurpose` 等），**须同步等待**返回后再继续
4. 子代理 prompt **开头**须含「语言要求」块（与 `subagent-inline-loop` 一致）：

```text
【语言要求】
- 全程使用中文：任务说明、执行过程、结论、返回报告
- 代码注释与公开 API 文档注释必须使用中文
- commit message 使用中文
- 仅代码标识符、协议字段名、第三方 API 等保持英文
```

5. 修复任务 prompt 须包含：

- 仓库/worktree 路径、分支名
- 探索结论（根因、影响文件、修复方向）
- 约束：单次 inline 完成；按逻辑块提交；运行针对性测试/构建；不 revert 无关改动

6. 子代理须按下列格式**中文**返回：

- 已完成项与剩余缺口
- 提交记录（sha + message），如有
- 验证命令及通过/失败
- 阻塞项（如有）

7. 若返回不合格或验证失败：修正 prompt 或补充探索后**重派**，不得口头宣称已修复

**阶段完成**：更新 `dynamic`（分支、HEAD_SHA、验证结果、阻塞项）与 `persist`（修复要点、主要提交）。

### 3. 补充 prd.md 与 spec.md（修复之后）

修复并验证通过后，**基于真实探索与改动**落盘文档（禁止编造未发生的修改）：

1. 写入 `bugs/<bug名称>/prd.md`（业务/现象视角）
2. 写入 `bugs/<bug名称>/spec.md`（技术/fix 视角）
3. 执行 `apm kb index rebuild`

**阶段完成**：更新 `dynamic`（prd/spec 路径、状态「待用户确认」）与 `persist`（bug 路径、根因摘要、修复摘要）。

### 4. 验证与收尾

- 主代理确认：针对性测试 + 项目约定 build（如 `npm run build`）已通过
- 向用户说明 bug 路径、根因、改动摘要，并请确认 `prd.md` / `spec.md`
- **用户确认后**：更新 `dynamic`（状态「bug 已修复」）与 `persist`（已确认的根因与修复结论）

## 探索指引

| 探索项 | 目的 |
|--------|------|
| 复现路径 / 日志 / 报错 | 确认现象与触发条件 |
| 调用链与状态 | 定位根因层级 |
| 相关测试 | 是否已有覆盖、如何防回归 |
| 父级 PRD / 历史 bugs | 是否已知问题或范围冲突 |
| 近期提交 | 是否为回归 |

```text

- 代码注释与公开 API 文档注释必须使用中文

- 仅代码标识符、协议字段名、第三方 API 等保持英文

请以 inline 模式（单次连续执行、不拆子代理）修复下列 bug。

仓库/worktree：<PATH>
分支：<BRANCH>
所属迭代：<需求名称>
Bug：<bug名称>

探索结论：
- 现象：…
- 根因：…
- 影响文件/模块：…
- 建议修复方向：…

约束：
- 单次运行内完成修复与验证
- 按逻辑块提交；不 revert 无关改动
- 运行与本 bug 相关的针对性测试/构建
- 新增/修改代码须写中文注释；公开 API 按项目语言补充文档注释

1）已完成项与剩余缺口
2）提交记录（sha + message）

```

## 文档格式规范

两文件均须以 **YAML Front Matter** 开头。

### prd.md

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `date` | string | 是 | `YYYY-MM-DD` |
| `dependency` | string \| string[] | 是 | 父级 PRD 路径（相对 `kb/docs/`）；无则 `[]` |

### spec.md

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `date` | string | 是 | `YYYY-MM-DD` |

## 文档模板

### prd.md

```markdown
---
date: YYYY-MM-DD
dependency: Iterations/<需求名称>/prd.md
---

# <bug名称> Bug PRD

## 背景

## 现象描述

## 复现步骤

## 预期行为

## 实际行为

## 影响范围

## 验收标准

## 回归测试要点
```

### spec.md

```markdown
---
date: YYYY-MM-DD
---

# <bug名称> Bug 修复规格（SPEC）

## 根因分析

## 修复方案

## 变更点清单

## 详细改动说明

## 测试策略
### 测试用例

## 风险与回滚方案
```

## 阶段记忆更新

| 阶段 | dynamic（当前任务） | persist（跨会话结论） |
|------|---------------------|---------------------|
| 探索完成 | 现象、根因、影响面、待修复项 | 根因要点、关键模块映射 |
| 修复完成 | 分支、HEAD_SHA、验证结果 | 修复要点、主要提交 |
| 文档落盘 | prd/spec 路径、待确认 | bug 路径、根因与修复摘要 |
| 用户确认 | 状态「bug 已修复」 | 已确认的根因与修复结论 |

## 执行检查清单

- [ ] 已 `apm read` / `apm kb search`，并完成代码探索、形成根因结论
- [ ] 探索后已更新 `dynamic` 与 `persist`
- [ ] 已在 `fix/<bug名称>`（或非保护分支）派子代理 inline 修复并同步等待
- [ ] 子代理返回含验证结果；针对性测试与 build 已通过
- [ ] 修复后已更新 `dynamic` 与 `persist`
- [ ] 已生成 `bugs/<bug名称>/prd.md` 与 `spec.md`（含 Front Matter）并已 `apm kb index rebuild`
- [ ] 文档内容与真实改动一致，未编造
- [ ] 文档落盘后已更新 `dynamic` 与 `persist`
- [ ] 已向用户说明并请确认文档；确认后已更新记忆

## 失败处理

**子代理调用失败**（如 `resource_exhausted`）：

1. 短暂停顿后重试一次
2. 仍失败：主代理可手工对照探索结论修复，但须同样运行验证并记录于 dynamic
3. 资源恢复后优先回到子代理流程

**无法复现或根因不明**：停止修复，在对话中说明缺口，用 `AskQuestion` 向用户索取信息；勿猜测性改代码。
