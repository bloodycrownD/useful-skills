# 敏捷开发

在**已有迭代**（`Iterations/<需求名称>/`）内做轻量、快速的代码交付：先摸清上下文，再派子代理改代码，**代码完成并验证通过后**在 `bugs/` 或 `features/` 下补充 `prd.md` 与 `spec.md` 留痕。

本 skill 特点：**先实现、后文档**，适合迭代内小步快跑。

主代理负责编排；**探索与实现类工作须派子代理完成**，主代理不得跳过子代理自行深入读代码或改代码（资源故障见「失败处理」）。

主代理与子代理的协作、说明、文档与回复**一律使用中文**（路径、标识符、协议字段等除外）。

---

## 适用范围

本 skill 面向**范围可控**的迭代内变更。下列情况须**停止**并在回复中说明原因，请用户缩小范围或换用其他方式：

- 无法定位根因或复现条件（bug）
- 变更明显跨多模块、改公共 API、与用户未确认的父级 PRD 验收冲突
- 用户要求的全维评审/正式 spec 驱动交付（超出本 skill 职责）

实现后 **最低质量闸**：针对性测试 + 项目约定 build 已通过。

---

## 总流程

```text
判定类型（bug / feature）→ 探索定位 →（必要时澄清）→ 子代理实现（同步等待）→ 验证
                                                              ↓
                                    补充 bugs|features/<敏捷名称>/prd.md + spec.md → 收尾
```

---

## 类型与路径

| 类型 | 何时使用 | 文档目录 | 推荐分支 |
|------|----------|----------|----------|
| **bug** | 缺陷、异常、回归、与预期不符 | `bugs/<敏捷名称>/` | `fix/<敏捷名称>` |
| **feature** | 迭代内小功能、范围变更、增强 | `features/<敏捷名称>/` | `feature/<敏捷名称>` |

### 路径约定

| 产物 | 路径 |
|------|------|
| PRD | `.apm/kb/docs/Iterations/<需求名称>/bugs\|features/<敏捷名称>/prd.md` |
| SPEC | `.apm/kb/docs/Iterations/<需求名称>/bugs\|features/<敏捷名称>/spec.md` |

- `<需求名称>`：所属迭代（与 `Iterations/` 下目录一致）；无法确定时询问用户
- `<敏捷名称>`：简短 kebab-case 标识（如 `login-timeout-on-refresh`、`export-csv-button`）
- 两个文档写完后均执行 `apm kb index rebuild`

### 输入来源

- 用户当轮描述（现象、诉求、变更点）
- **前文讨论结果**（对话中已拍板的结论、答疑要点）— 若对话中已有探索结论，主代理须将其纳入实现 prompt，**但不得跳过 Step 2 探索**

---

## Step 1：判定类型与准备

1. 根据用户意图判定 **bug** 或 **feature**；含糊时只问**一个**聚焦问题
2. 确定 `<需求名称>`、`<敏捷名称>`（kebab-case）
3. 执行 `apm read`，`apm kb search --q "<关键词/需求名称>"`
4. 读取父级 `.apm/kb/docs/Iterations/<需求名称>/prd.md`（及已有 `spec.md` 若存在）
5. **分支安全闸**：禁止在 `main` / `master` 直接改；工作区须干净

**阶段完成**：更新 `dynamic`（类型、敏捷名称、迭代名、分支计划）与 `persist`（父级 PRD 路径）。

---

## Step 2：探索定位（不可跳过）

探索由 **主代理编排、子代理执行**。

### 主代理职责

1. 将探索面拆为 **1–3 个**互不重叠子任务（bug 偏根因链；feature 偏影响面）
2. **并行** 派遣 readonly 探索子代理（`Task`，`subagent_type: explore`，`readonly: true`），**同步等待**全部返回
3. 汇总探索报告为「探索结论」：
   - **bug**：现象、预期/实际、触发条件、疑似根因、影响文件、修复方向、回归风险
   - **feature**：变更前现状、相对原范围差异、影响模块/接口、实现方向、风险
4. 关键信息缺失时，用 `AskQuestion` 只问**一个**聚焦问题

### 主代理禁止

- 自行深入读代码替代子代理
- 未等探索子代理返回即派实现或下结论

### 子代理派遣规范（探索）

| 项 | 值 |
|----|-----|
| 工具 | `Task`，`subagent_type: explore` |
| readonly | **true** |
| 并行 | 1–3 个，同步等待 |
| 失败 | 重试一次 → 手工探索，dynamic 标注 |

**阶段完成**：更新 `dynamic`（探索结论、待实现项）与 `persist`（根因/变更要点、关键模块映射）。

### 探索子代理须返回（探索报告）

```text
1）探索范围
2）发现摘要（与本次 bug/feature 直接相关）
3）证据列表（路径、符号、配置、文档）
4）建议实现/修复方向与风险
5）未弄清项（如有）
```

### 探索子代理 prompt 模板

```text
【语言要求】
- 全程使用中文；路径、符号、命令可保留原文

请以 readonly 模式探索下列范围，为敏捷开发提供事实依据（不修改任何文件）。

仓库：<REPO_PATH>
类型：<bug | feature>
所属迭代：<需求名称>
敏捷名称：<敏捷名称>
用户诉求 / 前文结论：<SUMMARY>
探索范围：<SCOPE>

请先列出将阅读的文件/目录，再按「探索报告」结构返回。
```

---

## Step 3：子代理实现（同步等待）

探索结论足以动手时：

1. 切换到推荐分支（`fix/...` 或 `feature/...`）或独立 worktree
2. 向子代理派发 **单个 inline 实现任务**（`Task`，`subagent_type: generalPurpose`，`readonly: false`），**须同步等待**返回
3. 子代理 prompt **开头**须含「语言要求」块（见下方 impl 模板）
4. 任务 prompt 须包含：仓库路径、分支、类型、探索结论、约束（单次 inline、按逻辑块提交、针对性测试/构建、中文注释与 commit）
5. 子代理须**中文**返回：已完成项、提交（sha + message）、验证结果、阻塞项
6. 返回不合格或验证失败：修正 prompt 或补探索后**重派**；同一问题重试 **≤3 次**，仍失败 → **blocked**，请用户拍板

### 子代理派遣规范（实现）

| 项 | 值 |
|----|-----|
| 工具 | `Task`，`subagent_type: generalPurpose` |
| readonly | **false**（单次 inline，禁止再拆子代理） |
| 失败 | 重试一次（资源）→ 修正 prompt 重派；震荡 ≥3 → blocked |

### 子代理实现 Prompt 模板

```text
【语言要求】
- 全程使用中文：任务说明、执行过程、结论、返回报告
- 代码注释与公开 API 文档注释必须使用中文
- commit message 使用中文
- 仅代码标识符、协议字段名、第三方 API 等保持英文

请以 inline 模式（单次连续执行、不拆子代理）完成下列敏捷开发任务。

仓库/worktree：<PATH>
分支：<BRANCH>
类型：<bug | feature>
所属迭代：<需求名称>
敏捷名称：<敏捷名称>

探索结论：
<粘贴汇总后的探索结论>

约束：
- 单次运行内完成实现与验证
- 按逻辑块提交；不 revert 无关改动
- 运行与本次改动相关的针对性测试/构建
- 新增/修改代码须写中文注释；公开 API 按项目语言补充文档注释

请严格按以下格式用中文返回：
1）已完成项与剩余缺口
2）提交记录（sha + message）
3）验证命令及通过/失败结果
4）阻塞项（如有）
```

**阶段完成**：更新 `dynamic`（分支、HEAD_SHA、验证结果）与 `persist`（实现要点、主要提交）。

---

## Step 4：验证

主代理确认（子代理已跑过仍须复核）：

- 针对性测试 + 项目约定 build（如 `npm run build`）已通过
- **bug**：现象已消除或达到用户可接受状态
- **feature**：变更点与探索阶段拍板范围一致

未通过则回到 Step 2 或 Step 3，不得进入留痕文档。

**阶段完成**：更新 `dynamic`（验证结果、是否通过）与 `persist`（验证命令摘要、通过/失败结论）。

---

## Step 5：留痕文档（实现之后）

实现并验证通过后，**基于真实探索与改动**落盘（禁止编造未发生的修改）：

1. 写入 `bugs/<敏捷名称>/` 或 `features/<敏捷名称>/` 下的 **`prd.md`**（业务/现象/需求视角）
2. 写入同目录 **`spec.md`**（技术/实现视角）
3. 执行 `apm kb index rebuild`

### prd.md Front Matter

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `date` | string | 是 | `YYYY-MM-DD` |
| `dependency` | string \| string[] | 是 | 父级 PRD：`Iterations/<需求名称>/prd.md`；无则 `[]` |

### spec.md Front Matter

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `date` | string | 是 | `YYYY-MM-DD` |
| `agile_trace` | boolean | 是 | 固定 `true`，标记后补留痕 spec |

### prd.md 模板（按类型取舍章节）

**bug**

```markdown
---
date: YYYY-MM-DD
dependency: Iterations/<需求名称>/prd.md
---

# <敏捷名称> Bug PRD

## 背景

## 现象描述

## 复现步骤

## 预期行为

## 实际行为

## 影响范围

## 验收标准

## 回归测试要点
```

**feature**

```markdown
---
date: YYYY-MM-DD
dependency: Iterations/<需求名称>/prd.md
---

# <敏捷名称> Feature PRD

## 背景与变更动机

## 范围说明（相对原需求）

## 影响模块与接口

## 验收标准

## 测试用例
```

### spec.md 模板（通用）

```markdown
---
date: YYYY-MM-DD
agile_trace: true
---

# <敏捷名称> 实现规格（SPEC）

## 根因 / 方案摘要

## 变更点清单

## 详细改动说明

## 测试策略
### 测试用例

## 风险与回滚方案
```

**阶段完成**：更新 `dynamic`（prd/spec 路径、状态「待用户确认」）与 `persist`（敏捷项路径、摘要）。

---

## Step 6：收尾

- 向用户说明：类型、敏捷名称、路径、探索结论、改动与验证摘要
- 请用户确认 `prd.md` / `spec.md` 与代码一致
- **用户确认后**：更新 `dynamic`（状态「敏捷项已完成」）与 `persist`（已确认摘要）

---

## 阶段记忆更新

| 阶段 | dynamic | persist |
|------|---------|---------|
| 准备 | 类型、敏捷名称、迭代、分支计划 | 父级 PRD 路径 |
| 探索完成 | 探索结论、待实现项 | 根因/变更要点、模块映射 |
| 实现完成 | 分支、HEAD_SHA | 实现要点、主要提交 |
| 验证完成 | 验证结果、是否通过 | 验证命令摘要、通过/失败结论 |
| 文档落盘 | prd/spec 路径、待确认 | 敏捷项路径、摘要 |
| 用户确认 | 状态「敏捷项已完成」 | 已确认摘要 |

## 执行检查清单

- [ ] 已判定 bug / feature，并确定需求名称与敏捷名称
- [ ] 已读父级 PRD，`apm read` / `apm kb search`
- [ ] 已派遣 readonly 探索子代理并同步等待；主代理已汇总探索结论
- [ ] 已在非保护分支派子代理 inline 实现并同步等待
- [ ] 针对性测试与 build 已通过
- [ ] 已生成 `bugs|features/<敏捷名称>/prd.md` 与 `spec.md`（含 Front Matter）并已 `apm kb index rebuild`
- [ ] 文档内容与真实改动一致，未编造
- [ ] 已向用户说明并请确认；确认后已更新 dynamic / persist

## 失败处理

**子代理调用失败**（如 `resource_exhausted`）：

1. 短暂停顿后重试一次
2. 仍失败：主代理可手工完成，但须同样运行验证并记录于 dynamic
3. 资源恢复后优先回到子代理流程

**无法定位或范围不清**：

- **bug** 无法复现/根因不明：停止改代码，说明缺口，`AskQuestion` 索取信息
- **feature** 与原迭代范围冲突：先与用户确认范围，再实现

**勿猜测性改代码。**
