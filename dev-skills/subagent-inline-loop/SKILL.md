# Subagent Inline Loop

## 目的

在 **spec 为唯一事实来源** 的前提下，通过「拆解 → 编排 → 派遣 → 收敛」完成实现：

1. 将 spec 拆解为带依赖的任务节点（DAG）
2. 按拓扑顺序编排执行波次（可并发、可串行）
3. 向子代理派遣任务，**同步等待**其完成后再推进
4. 汇总结果；若未达 merge-ready，更新 DAG 并进入下一轮

主代理负责编排与调度，**不得**跳过子代理自行完成实现类任务（资源故障见「失败处理」）。

主代理与子代理的协作、说明、评审结论**一律使用中文**（代码标识符、协议字段、第三方 API 名称等除外）。

---

## 关键术语

### inline

子代理在**单次连续运行**内完成所派任务（实现、验证、提交等），不在子代理内部再拆多层子代理。

### 子代理

通过 `Task` 等机制派发的独立执行单元。本 skill **不绑定**具体子代理类型；按任务性质选用合适类型即可。

### 同步等待

- 每波次内：所有可并发节点派发后，**须等待全部返回**再进入下一波次
- 单节点：派发后**须等待**该子代理结束，再消费其输出、更新状态
- 禁止「发了不等」或仅凭假设推进

### DAG 任务图

拆解后的任务节点及依赖关系：

- **节点**：一项可独立派发给子代理的工作单元（实现、测试、评审等）
- **边（依赖）**：`B` 依赖 `A` 表示 `A` 完成后才能启动 `B`
- **波次（wave）**：同一波次内节点彼此无未完成依赖，可并行派遣

---

## 开始前检查（必须满足）

- 目标 worktree/branch 工作区干净（无未提交改动）
- 已知：`Spec path`（唯一事实来源）、`PRD path`（可选）、`Repo/worktree path`、当前分支名
- 执行 `apm read`，并用 `apm kb search --q "<需求名称/spec 关键词>"` 检索相关文档
- 分支安全闸：禁止在 `main` / `master` 直接开发；先切功能分支或独立 worktree（推荐 `feature/<topic>`、`fix/<topic>`）

工作区不干净时先 commit 或 stash；位于保护分支时先完成分支切换。

**准备完成**：更新 `dynamic`（spec/PRD 路径、分支、轮次 0）与 `persist`（需求名称、spec 路径、分支名）。

---

## 总流程

```text
准备 → 拆解 DAG → 编排波次 → [按波次派遣子代理并同步等待] → 汇总评审
                                                              ↓
                                                    merge-ready? ─否→ 更新 DAG / must-fix → 下一轮
                                                              ↓是
                                                    最终验证 → 结束
```

---

## Step 1：任务拆解（生成 DAG）

基于 spec 将工作拆为节点，至少覆盖：

| 节点类型 | 说明 | 典型依赖 |
|----------|------|----------|
| 实现 | 按 spec 章节或模块落地代码 | 无，或依赖前置调研/契约节点 |
| 测试/验证 | 运行针对性测试、构建、文档校验 | 依赖对应实现节点 |
| 评审 | 对照 spec 检查完成度与质量 | 依赖本轮相关实现 + 测试节点 |

拆解原则：

- 粒度：一个节点对应子代理**单次 inline**可完成的闭环工作
- 依赖：共享文件、接口契约、数据模型等须显式建边，避免并发写冲突
- 可并行：无依赖且资源不冲突的节点放入同一波次
- 必须包含至少一个**评审节点**作为收敛闸门（输出 `merge-ready` / `not merge-ready`）

拆解产出（主代理维护，可写入 `dynamic`）示例：

```text
节点 id | 类型   | 描述           | 依赖
impl-a  | 实现   | 模块 A 按 spec | -
impl-b  | 实现   | 模块 B 按 spec | -
test    | 验证   | 针对性测试构建 | impl-a, impl-b
review  | 评审   | spec 符合性    | test
```

---

## Step 2：任务编排

在派遣前完成编排，不可跳过：

1. **校验 DAG**：无环；依赖指向的节点均存在
2. **拓扑排序**：划分执行波次 `wave-0, wave-1, …`
3. **并发策略**：
   - 同波次、无文件/分支冲突的节点 → 同一轮**并行**派发
   - 有依赖 → **串行**等待上游波次全部完成
   - 不确定是否冲突时，优先串行或拆波次
4. **输入绑定**：为每个节点准备 prompt 上下文（spec 路径、repo、分支、上游节点产出、上一轮 must-fix）；**prompt 正文须为中文**，并显式写明语言要求（见「中文要求」）
5. **记录编排**：当前波次、待执行节点、进行中/已完成状态写入 `dynamic`

**编排完成**：更新 `dynamic`（DAG 摘要、波次计划、当前 wave）与 `persist`（本轮任务范围、关键依赖）。

---

## Step 3：按波次派遣子代理（同步等待）

对每个波次依次执行：

1. 将该波次内所有就绪节点**一并派发**给子代理（可并行多个 `Task`）
2. **同步等待**本波次全部子代理返回
3. 收集并校验输出（见「子代理输出契约」）
4. 若有节点失败或阻塞：记录原因，决定是否重试、降级或中止本轮
5. 本波次全部成功后，更新 `dynamic`，再进入下一波次

### 派遣约束（写入子代理 prompt）

每条 prompt **开头**须包含「中文要求」块（见下方模板），再写任务细节：

- inline：单次运行完成所派任务，不再拆子代理
- **全中文执行**：思考、说明、总结、评审、与用户可见输出均用中文；commit message 用中文
- 按逻辑块提交；不回滚无关改动
- 运行与节点相关的针对性测试/构建
- 新增/修改逻辑的注释**必须为中文**（边界、校验、状态转换）；禁止逐行翻译式注释

### 中文要求（每条子代理 prompt 必含）

```text
【语言要求】
- 全程使用中文：任务说明、执行过程、结论、评审意见、返回报告
- 代码注释与公开 API 文档注释（JSDoc/TSDoc、JavaDoc、Python docstring 等）必须使用中文
- commit message 使用中文
- 仅代码标识符、协议字段名、第三方 API 等保持英文
```

### 子代理输出契约

所有返回内容**须为中文**（路径、sha、命令行等技术片段可保留原文）。

实现/验证类节点须返回：

- 已完成项与剩余缺口
- 提交（sha + message），如有
- 验证命令及 通过/失败
- 阻塞项（如有）

评审类节点须返回：

- 按严重级别排序的问题（含文件引用）
- 对照 spec 的完成矩阵
- 必改项（must-fix）列表
- 结论：`可合并（merge-ready）` 或 `不可合并（not merge-ready）`

**每轮全部波次执行完毕**：更新 `dynamic`（轮次、HEAD_SHA、评审结论、must-fix）与 `persist`（已实现要点、未解决风险）。

---

## Step 4：收敛决策

- **可合并（merge-ready）**：
  - 执行最终验证（覆盖改动范围的测试 + `npm run build`；若涉及 web：`npm --prefix web run build`）
  - 用中文更新 `dynamic` / `persist`，并以中文给出集成选项（PR / merge / cleanup）
- **不可合并（not merge-ready）**：
  - 提取必改项，更新 DAG（新增修复节点、调整依赖，或重跑受影响子图）
  - 轮次 +1，回到 Step 2 重新编排后执行

---

## 开发规范：中文注释与文档注释

实现类节点须遵守；**按项目语言选用该生态的文档注释风格**，不强行统一为某一种格式。

### 注释分层

| 层级 | 范围 | 要求 |
|------|------|------|
| 行内/块注释 | 实现细节、分支逻辑 | 中文；解释「为什么」，禁止逐行翻译代码 |
| 模块/文件头 | 文件、类、包 | 中文；说明职责边界、关键不变量、调用关系 |
| 公开 API 文档注释 | 对外导出的函数、类、方法、类型 | 中文；说明用途、参数、返回值、异常/副作用 |

### 按语言的文档注释风格（择项目所用）

| 语言/生态 | 推荐风格 | 说明 |
|-----------|----------|------|
| TypeScript / JavaScript | JSDoc / TSDoc | `@param` `@returns` `@throws` 等标签可用，描述正文中文 |
| Java / Kotlin | JavaDoc / KDoc | `/** … */`；`@param` `@return` `@throws` |
| Python | docstring | 模块/类/函数使用三引号文档串；公开 API 说明参数与返回值 |
| Go | godoc | 导出符号上方连续 `//` 注释，首句为摘要 |
| Rust | `///` / `//!` |  crate/模块/公开项文档注释 |
| C# | XML 文档注释 | `/// <summary>` 等，正文中文 |

无项目约定时：优先采用该语言社区最常见的文档注释形式；**内容用中文**，标签/结构跟生态习惯。

### 必写范围

1. **模块头**：负责范围、不负责什么、关键不变量、主要调用链路
2. **关键分支**：幂等、防重、资源上限、事务/序列化边界及原因
3. **公开 API**：参数含义、返回值、副作用、可能抛出的异常；核心对外接口须有完整文档注释

### 评审门槛

- 新增/修改代码存在英文说明性注释 → **必改项**
- 公开 API 缺少文档注释或仅有类型签名 → **必改项**
- 缺少关键语义注释、难以理解 → **必改项**

## 阶段记忆更新

| 阶段 | dynamic | persist |
|------|---------|---------|
| 准备完成 | spec/PRD、分支、轮次 0 | 需求名称、spec 路径、分支 |
| 拆解/编排完成 | DAG 摘要、波次计划、当前 wave | 任务范围、关键依赖 |
| 每轮结束 | 轮次、HEAD_SHA、评审结论、must-fix | 已实现要点、未解决风险 |
| merge-ready | 状态、验证结果、集成选项 | 完成摘要、主要提交、遗留项 |

探索/准备期间：`apm read` + `apm kb search`。

---

## 验证规则（禁止口头通过）

- 各验证节点：至少覆盖其上游改动的最小测试集
- 涉及契约/文档：运行项目内等效覆盖率/文档校验
- 宣布 merge-ready 前：根测试 + 全量 build（按项目约定）

---

## 失败处理

**子代理调用失败**（如 `resource_exhausted`）：

1. 短暂停顿后重试该节点一次
2. 仍失败：主代理可用中文做手工 spec 对照（映射验收点 → 代码 + 测试证据），整理必改项
3. 资源恢复后回到子代理编排循环；手工介入后同样更新 `dynamic` / `persist`

**节点输出不合格**：不进入依赖它的下游波次；修复后从该节点或对应子图重跑。

## Prompt 模板

以下模板均为中文；派遣时逐字保留「语言要求」块，再填入具体任务信息。

### 实现类节点

```text
【语言要求】
- 全程使用中文：任务说明、执行过程、结论、返回报告
- 代码注释与公开 API 文档注释（JSDoc/TSDoc、JavaDoc、Python docstring 等）必须使用中文
- commit message 使用中文
- 仅代码标识符、协议字段名、第三方 API 等保持英文

请以 inline 模式（单次连续执行、不拆子代理）完成下列任务。

仓库/worktree：<PATH>
分支：<BRANCH>
Spec：<SPEC_PATH>
PRD（可选）：<PRD_PATH>
本节点任务：<NODE_DESCRIPTION>
上游节点产出：<...>
上一轮评审必改项：
- ...

约束：
- 单次运行内完成任务；按逻辑块提交
- 运行与本任务相关的针对性测试/构建
- 不 revert 无关改动
- 新增/修改代码须写中文注释；公开 API 按项目语言补充文档注释（JSDoc/JavaDoc/docstring 等）

请严格按以下格式用中文返回：
1）已完成项与剩余缺口
2）提交记录（sha + message）
3）验证命令及通过/失败结果
4）阻塞项（如有）
```

### 评审类节点

```text
【语言要求】
- 全程使用中文：评审过程、问题描述、结论、返回报告
- 仅文件路径、sha、命令行等技术片段可保留原文

请对照 spec 评审本轮已完成工作是否符合要求。

仓库/worktree：<PATH>
Spec：<SPEC_PATH>
BASE_SHA：<SHA>
HEAD_SHA：<SHA>
本轮已完成节点：<...>

请严格按以下格式用中文返回：
1）按严重程度排序的问题（严重 / 重要 / 轻微），含精确文件引用
2）对照 spec 章节的完成矩阵
3）必改项（must-fix）列表
4）结论：可合并（merge-ready）或 不可合并（not merge-ready）
```

### 验证类节点

```text
【语言要求】
- 全程使用中文：执行说明、失败分析、返回报告
- 仅命令行、日志片段等技术内容可保留原文

请对上游实现节点的产出运行验证。

仓库/worktree：<PATH>
Spec：<SPEC_PATH>
验证范围：<上游涉及的文件/模块>

请严格按以下格式用中文返回：
1）执行的命令及通过/失败结果
2）失败项详情（含日志/片段）
3）下游评审是否可以继续（是/否及原因）
```

## 备注

- 文档默认位于 `.apm/kb/docs/Iterations/<需求名称>/`（`prd.md`、`spec.md`）
- 实现偏离 spec 时，**先更新 spec** 再改 DAG 与实现
- 优先小而可回滚的提交；同文件多节点默认串行或分波次，避免并发冲突
- 子代理 prompt 禁止仅用英文撰写；未含「语言要求」块的派遣视为不合规
